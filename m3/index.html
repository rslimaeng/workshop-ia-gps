Uso valioso · pouco explorado

Validação de Premissas

Um dos usos mais poderosos da IA em orçamento — e o menos usado: pedir que ela questione as premissas do seu orçamento antes de você enviar pro cliente. Toda obra GPS tem premissas que parecem óbvias internamente mas são exatamente onde o orçamento estoura.

Prompt base — Validação de Premissas 

Papel: Você é um engenheiro de obras com experiência em orçamentação de projetos
em Fortaleza/CE.

Contexto: Estou finalizando um orçamento de [tipo de projeto] e quero validar
as premissas antes de enviar para o cliente.

Tarefa: Analise as premissas abaixo e:
1. Identifique premissas que tipicamente geram erros ou subdimensionamento
2. Aponte o que pode estar faltando para este tipo de projeto em Fortaleza
3. Sinalize premissas que precisam de mais dados para ser validadas

Premissas do projeto:
[cole as premissas aqui]

Formato: lista numerada com classificação:
✓ Adequada / ⚠️ Verificar / ✗ Crítica
Quando usar: antes de cada envio de orçamento ao cliente, especialmente em obras com prazo apertado ou cliente novo. 10 minutos de validação aqui podem economizar semanas de aditivo depois.

76% da turma reconhece que poderia ser mais ágil ao comunicar para gestores e clientes. Usar IA acelera a produção — mas pode gerar desconfiança se não for comunicado bem.

Comunicação executiva

Da Análise à Decisão: OIA + Regra do Headline

Você analisou o orçamento com IA. Tem números, gráficos, planilhas. Agora vem a parte mais difícil: o cliente lê os títulos dos slides em 30 segundos e decide se confia em você. Dado sem narrativa não decide nada — e narrativa sem framework cansa quem está lendo.

OIA — Observação → Insight → Ação

Todo dado tem 3 camadas. A maioria dos orçamentos para no primeiro:

Nível ❌ Exemplo ruim ✅ Exemplo OIA (orçamento GPS)
O — Observação
O que os dados mostram?		 "Orçamento total: R$ 280k" "Orçamento total: R$ 280k — 22% acima do similar de 2025"
I — Insight
O que isso significa?		 — "O aumento vem majoritariamente de insumos (+18% SINAPI no aço CA-50) — fora do controle da equipe"
A — Ação
O que fazer?		 — "Apresentar 2 cenários: prazo padrão (R$ 280k) e prazo otimizado com lote único de compra (R$ 265k)"
Regra de ouro: entregar só a Observação é deixar o cliente sozinho com a parte mais difícil — ele tem o dado, mas não sabe o que ele significa nem o que fazer. OIA garante que você entrega as três camadas no lugar que importa.

A Regra do Headline de Insight

Cliente e gestor escaneiam os títulos antes de ouvir você falar. Se o título já é o insight, você já ganhou a atenção antes da primeira palavra. Se é só um rótulo ("Análise do Orçamento"), eles esperam você explicar — e podem perder o fio.

❌ Título de dado (rótulo) ✅ Título de insight
Resultados do Orçamento Aumento de 22% vem de insumos SINAPI — não de mão de obra
Análise por Categoria Estrutural pesa 41% do total — única alavanca real de negociação
Comparativo com 2025 Mão de obra está em linha; o aço CA-50 puxou R$ 38k
Próximos Passos Cotar 3 fornecedores de aço antes de fechar — economia potencial R$ 15k
Teste prático: leia só os títulos do seu deck em sequência, sem abrir nenhum slide. Se problema, causa e o que fazer ficam claros só pelos títulos, você acertou. Se você precisa abrir o slide pra entender, ainda é rótulo.

Prompt base — Narrativa OIA para Orçamento 

Papel: Você é um engenheiro especialista em comunicação executiva de
orçamentos para clientes não-técnicos.

Contexto: Estou preparando a apresentação de um orçamento de [tipo de
obra, valor total, comparação com referência] para [cliente / gestor].
Os dados estão abaixo:
[cole o resumo do orçamento — itens principais, valores, comparativos]

Tarefa: Estruture uma narrativa executiva seguindo o framework OIA
(Observação → Insight → Ação) para 5 a 7 pontos do orçamento. Cada
ponto deve ter um título que já é um insight — não um rótulo.

Formato: tabela com 3 colunas — Título do slide | Observação (dado) |
Ação recomendada (decisão). Máximo 7 pontos.

Limitações: Não use linguagem comercial. Não invente dados que não
estão no resumo. Se faltar contexto pra concluir um insight, sinalize.

Critério de Sucesso: o cliente lê só os títulos em sequência e entende
onde estão os maiores valores, por que estão lá e o que fazer.

Negativo: NÃO use títulos genéricos como "Resultados" ou
"Próximos Passos" sem conteúdo no próprio título.

Processos · A raiz do output genérico

Quando o Processo Não Muda, o Output Parece IA

No M2 vimos que IA roda sobre trilhos — e trilhos vagos produzem respostas genéricas. Há um nível mais profundo: se o processo que você executa todo dia não foi redesenhado para trabalhar com IA, o output vai carregar a marca da ferramenta, não da GPS. A revisão OIA resolve a narrativa. O IPO resolve a raiz.

O Framework IPO (Input → Processo → Output) abre a caixa preta de um processo recorrente e o reconstrói com critérios claros — o que a IA precisa para executar com consistência e entregar no seu jeito, não no jeito genérico.

Input O que aciona o processo dado, documento ou solicitação que chega
→
Processo 4–7 passos com verbo + critério a caixa preta aberta — cada passo é uma instrução
→
Output Formato e destino exatos o resultado pronto — para quem e em qual formato

M3 · Processos

Framework IPO — Input · Processo · Output

Abre a caixa preta de um processo recorrente e o reconstrói com instruções que a IA executa com consistência. Conecta com os trilhos do M2 — e prepara o terreno para o Assistente Persistente do M5.

Input O que aciona o processo
  • Dado que chega (formulário, e-mail, planilha)
  • Formato exato dos dados de entrada
  • Quem aciona e com qual frequência
→
Processo 4–7 passos com verbo + critério
  • Cada passo: verbo de ação + critério de decisão
  • Ex: "Se valor ≤ R$2k → aprovar; se > R$2k → encaminhar"
  • Passo vago = instrução que a IA vai inventar
→
Output Formato e destino exatos
  • Formato final entregue (e-mail, planilha, doc)
  • Destinatário definido
  • Critério de qualidade do resultado
+ Exceção / Erro — O que fazer quando algo faltar ou sair do fluxo normal. Um processo sem regra de fallback vai travar ou inventar na primeira situação inesperada. Ex: "se campo vazio → solicitar reenvio", "se sem resposta em 24h → escalar".

O princípio crítico

❌ Intenção vaga

"Verificar orçamento"

A IA não sabe o critério — vai inventar um razoável e o output vai parecer genérico.

✓ Instrução com critério

"Se valor ≤ alçada → aprovar; se > alçada → encaminhar ao Gestor com justificativa"

A IA executa sem ambiguidade — o output carrega o critério da GPS, não um genérico.

Os 6 campos da tabela IPO

Campo O que colocar Erro comum
Área Nome específico do processo Só o setor ("Administrativo") sem nomear o processo
Input O que aciona e o formato exato dos dados "Uma solicitação" sem detalhar o formato
Processo 4–7 passos com verbo + critério de decisão Passos vagos como "aprovar" sem o critério
Output Formato final e destinatário exatos "Resposta" sem definir formato ou destino
Exceção Regra de fallback para situações fora do padrão Deixar vazio — o resultado vai inventar ou travar
Recursos Dados, arquivos ou acessos necessários Listar sistemas sem API ou acesso disponível

Conexão com os módulos

↩ M2 — Trilhos: o IPO é o método para redesenhar os trilhos processo a processo. Trilho genérico = output genérico, independente do prompt.
↪ M5 — Assistente: o IPO preenchido vira o briefing do Assistente Persistente. Quanto mais preciso, menos cara-de-IA no output final. 
Papel: Você é um consultor de processos especializado em sistematização de tarefas repetitivas para automação ou criação de assistentes especializados.

Contexto: Eu (gestor na GPS Engenharia) tenho um processo que executo recorrentemente: [descreva — ex: análise semanal de obras, fechamento mensal de medições, briefing de fornecedores]. Quero mapear no formato IPO para depois transformar em Assistente Persistente.

Descrição do processo:
[passos que executo hoje, ferramentas que uso, quanto tempo gasto, frequência]

Tarefa: Mapeie o processo em IPO completo:
- INPUTS: o que precisa estar disponível antes de começar (dados, arquivos, contexto)
- PROCESS: passos sequenciais (com critério de "pronto" em cada um)
- OUTPUTS: o que sai pronto para usar (formato, destinatário, critério de qualidade)

Termine com 3 perguntas para definir qual parte vale automatizar com Assistente (Autopiloto), qual vale Colaboração e qual deve continuar Manual.

Formato: 3 seções (Inputs / Process / Outputs) seguidas das 3 perguntas. Cada passo do Process em 1 linha clara.

Limitações: Use apenas a descrição. Não invente passos que não foram mencionados. Se um passo estiver vago, peça detalhe antes de mapear.

Critério de Sucesso: posso pegar o output e usar diretamente como Instrução de Sistema para um Claude Project ou Custom GPT na semana seguinte.

Negativo: NÃO listar "etapa 1: planejar" sem conteúdo. NÃO confundir Input com Output. NÃO recomendar automatizar tudo — manter Manual o que exige julgamento.
